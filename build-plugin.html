Plugin kinds

Four kinds of block, one API

Every plugin is a block with a typed contract — it composes into the same rails, panels and canvas the built-in blocks use.

Visual

A new chart or visual that renders on the canvas from Calc-Engine results — bind fields, draw, react to cross-filters.

Connector

A data source that emits a governed dataset — the platform owns the connection, credentials and query, not your code.

Processor

A transform step in the data editor — take a dataset in, return a transformed dataset, fully reproducible.

UI block

A panel or surface in the rails and docks — present controls, settings and documentation alongside the rest of the app.

Locked-platform model

Untrusted by default, sandboxed by design

A plugin never touches databases, credentials, secrets or other plugins directly. Every request flows through the SDK and a permission check before the platform core returns governed data.

  • Declared permissions — granted at install, enforced at every call.
  • Isolated runtime — no shared memory, no reaching into other plugins.
  • Network off by default — outbound access is allow-listed per plugin.
  • Audited — installs, permission grants, data requests and exports are logged.

The manifest

Declare what your plugin needs

Every plugin ships a manifest. Permissions are reviewed at install — only granted ones are honoured at runtime.

// mybi-plugin.json
{
  "id":      "acme.sales-funnel",
  "name":    "Sales Funnel",
  "version": "1.0.0",
  "kind":    "visual",            // visual · connector · processor · ui
  "sdk":     "^1",
  "permissions": [
    "dataset.read",            // read governed datasets only
    "dashboard.widget.create"  // place a visual on the canvas
  ],
  "network": false                // outbound network off by default
}
Build & run

From scaffold to the canvas

  1. ScaffoldStart from the SDK template — manifest, entry point and a typed contract for your block kind.
  2. Develop in dev modeEnable Developer options in Settings → Project, load your plugin, and iterate live in the app.
  3. BuildBundle and minify your plugin into a single distributable artifact.
  4. Self-sign & shareSign it with your own key to ship as a community plugin anyone can install.

Self-verify before you ship

MyBI can verify any installed plugin's signature locally — confirm your build is intact and untampered before distributing it.

# build + sign (community tier)
pnpm build
mybi-sign ./dist --key ~/.mybi/dev.key

# verify locally
mybi-verify ./dist
✓ signature valid · acme.sales-funnel@1.0.0

Ready for the “Trusted by MyBI” badge?

Community plugins install freely. To ship as Trusted by MyBI, get your publisher key countersigned — no private key ever leaves your machine.
